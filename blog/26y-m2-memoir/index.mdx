## 들어가며

2월은 AI로 빠르게 해결할 수 없었던 에러를 만났던 즐거운 한 달이었다.
설날 연휴에 터진 결제 장애부터, 브라우저 간 인증 연동 버그까지 — 예상치 못한 문제들을 만났고, 그때마다 의도적으로 더 깊이 파고들었다. AI 덕분에 본질을 빠르게 파악할 수 있어서 유용했지만, 내가 파악한 본질이 정말 본질이 맞는지 다시 점검하는 태도도 필요하다는 걸 느꼈다.

돌이켜보면, 그 과정에서의 고민들이 2월의 가장 큰 자산이 된 것 같다.

> 2월에는 크게 세 가지 사건들이 있었다.

1. **설날 결제 장애 대응** — 사파리에서 결제 완료 페이지가 흰 화면으로 뜨는 문제를 긴급 대응했다. 임시 조치부터 근본 원인까지 파고든 과정을 나누려고 한다.
2. **UI 제안으로 문의율 제로 만들기** — 개발자가 먼저 고객 불편을 발견하고, 작은 UI 변경으로 웹 문의를 없앤 경험을 했고 사소할 수 있지만 의미있게 다가왔다.
3. **웹-지도 인증 연동 트러블슈팅** — 사파리/크롬에서 회원가입 후 지도에 인증이 반영되지 않던 문제. 크로스 브라우징 이슈를 제대로 만났었다.

### 1. 설날에 갑자기 리포트 결제가 안된다는 문의가!?

> "우리 오픈닥터의 리포트는 무엇인가?"를 먼저 공유하고자 한다.

오픈닥터 리포트 서비스는 사용자가 지정한 입지의 의원 정보를 조합해 개원에 도움이 되는 인사이트를 제공하는 유료 서비스다. 오픈닥터 웹에서 유일한 유료 서비스이자, 제품팀의 핵심 먹거리 중 하나다.

그런데 하필 설날 연휴에 "리포트를 결제했는데 안 보인다"는 문의가 빗발쳤다. 긴급 대응이 필요했고, 두 가지 방식으로 대응했다.

![img alt="customer-reception"](/img/study/report-response.png)

- 결제했지만 리포트를 못 본 고객 → **환불 처리**
- 반드시 리포트를 봐야 하는 고객 → **어드민으로 접속해 해당 지역 리포트를 PDF로 뽑아 메일로 직접 전달**

#### 그런데 왜 갑자기 안 된 거지?

중요한 맥락이 있다. 오픈닥터 웹은 Flutter-web(레거시)에서 Next.js로 점진적 마이그레이션을 진행 중이다.

- as-is
  ![img alt="m-report-asis"](/img/study/m-report-asis.png)
- to-be
  ![img alt="m-report-tobe"](/img/study/m-report-tobe.png)

리포트 상세 페이지를 Flutter-web에서 Next.js로 마이그레이션하고 배포한 뒤, 며칠 후 이 문제가 발견됐다. 그동안도 발생하고 있었겠지만 문의가 들어오지 않았다.

찾아보니 원인은 결제 모듈(iamport.js)에 있었다.

결제 흐름은 이렇다: **결제 완료 → 결제 완료 페이지 이동 → 서버에 "결제됐다" 알림 → 리포트 열람 가능.**

그런데 이 흐름에서 "결제 완료 페이지로 이동"하는 코드 한 줄이 빠져 있었다. 결제는 성공했지만 서버는 결제 사실을 모르는 상태. 고객 입장에서는 돈은 나갔는데 리포트가 안 열리는 상황이 된 것이다.

```js
// 추가된 코드 (fix)
window.parent.postMessage({ type: "paymentResult", ... }, "*");
window.location.href = uri; // 이 한 줄이 빠져있었다
```

크롬에서는 `postMessage`만으로도 결제 프로세스가 정상 동작했지만, 사파리에서는 결제 완료 페이지가 흰 화면으로 표시되며 프로세스가 끊겼다.

#### 배운 점

- 마이그레이션 과정에서 **두 시스템(레거시/신규) 사이의 통신 경로를 빠짐없이 추적**해야 한다. 한쪽만 확인하면 놓친다.
- 사파리와 크롬의 동작 차이는 직접 겪어보지 않으면 알기 어렵다. **크로스 브라우저 QA는 선택이 아니라 필수**다.

### 3. 오픈닥터 웹에서 인증하면 지도에 반영이 안 된다?

뉴오픈닥터(Next.js)에 회원가입 기능을 배포한 뒤, 며칠 후 발견된 문제였다. Next.js에서 회원가입/로그인을 하면 웹은 정상인데, **Flutter-web 지도에는 인증 상태가 반영되지 않았다.** 분명 QA를 했는데 놓쳤다. 크롬에서는 문제가 없었기 때문이다.

#### 구조 이해: Flutter-web에서 Next.js로 점진적 마이그레이션

우리 서비스는 Next.js가 부모이고 Flutter-web이 iframe으로 들어가는 구조다.
두 앱 사이의 인증 상태 동기화가 핵심인데, 이 부분이 빠져 있었다.

![img alt="m-auth-token-error"](/img/study/m-auth-token-error.png)

#### 근본 원인: Safari iframe에서 `FlutterSecureStorage`가 hang

디버깅 과정에서 알게 된 원인은 이랬다.

1. Next.js에서 회원가입 → `postMessage`로 Flutter-web에 토큰 전달
2. Flutter-web이 토큰을 `FlutterSecureStorage`에 저장 시도
3. `FlutterSecureStorage`의 웹 구현체는 내부적으로 **WebCrypto + IndexedDB**를 사용
4. Safari의 **ITP(Intelligent Tracking Prevention)**가 iframe 내 third-party IndexedDB 접근을 파티셔닝 처리
5. `crypto.subtle` 접근이 차단되면 Promise가 resolve/reject 없이 **무한 대기(hang)**
6. 결과: 토큰 저장 실패 → API 호출 시 인증 헤더 없음 → 지도에 인증 상태 미반영

크롬은 iframe 내 third-party storage 접근을 Safari만큼 엄격히 제한하지 않기 때문에, 크롬에서는 문제가 재현되지 않았다.

#### 해결: `FlutterSecureStorage` → 인메모리 토큰으로 전환

![img alt="m-auth-token-solve"](/img/study/m-auth-token-solve.png)

모든 곳에서 `FlutterSecureStorage` 직접 읽기를 제거하고, `AuthManager().opn_token`에 인메모리로 저장/읽기하도록 변경했다.

```dart
// Before: Safari iframe에서 hang 발생
const storage = FlutterSecureStorage();
var opnToken = await storage.read(key: "OPN_OPNDOCTOR_KEY_AUTH_OPN_TOKEN");

// After: 인메모리 토큰 사용 (Safari iframe에서 FlutterSecureStorage가 hang할 수 있어 사용하지 않음)
var opnToken = AuthManager().opn_token;
```

동시에 Next.js 쪽에서도 Flutter ↔ Next.js 간 인증 상태 양방향 동기화를 위한 메시지 핸들러(`useAuthMessageHandlers`)와 세션 관리 서비스(`authSession.ts`)를 새로 만들어 한 곳에서 관리하도록 리팩터링했다.

#### 더 깊이 파보기: FlutterSecureStorage는 내부에서 뭘 하고 있었나?

문제를 해결한 뒤에도 "왜?"가 남았다. `FlutterSecureStorage`가 내부에서 정확히 뭘 하길래 Safari에서 hang이 걸리는 건지 직접 파봤다.

우리 프로젝트의 버전은 `flutter_secure_storage_web` **v1.2.1**이었다. 이 버전은 내부적으로 **WebCrypto API + IndexedDB**를 사용한다.

##### FlutterSecureStorage 웹 구현체의 동작

```
write("token", "abc123") 호출 시:
  1. crypto.subtle로 AES-GCM 256비트 암호화 키 생성
  2. 12바이트 랜덤 IV(초기화 벡터) 생성
  3. AES-GCM으로 "abc123" 암호화
  4. "base64(IV).base64(암호문)" 형태로 IndexedDB에 저장

read("token") 호출 시:
  1. IndexedDB에서 암호화된 값 조회
  2. IV와 암호문 분리
  3. crypto.subtle로 AES-GCM 복호화
  4. 평문 반환
```

여기서 두 가지 핵심 기술이 등장한다.

**WebCrypto API (`crypto.subtle`)** — 브라우저 내장 암호화 인터페이스다. `generateKey()`, `encrypt()`, `decrypt()` 등의 메서드로 AES, RSA 등의 암호화를 수행한다. 단, **HTTPS(Secure Context)에서만 사용 가능**하고, HTTP에서는 `crypto.subtle`이 `undefined`가 된다.

**IndexedDB** — 브라우저 내장 비동기 데이터베이스다. localStorage보다 용량이 크고, 비동기 API라 메인 스레드를 블로킹하지 않는다. 문제는 이 "비동기"라는 특성이 Safari ITP와 만나면 치명적이라는 것이다.

##### Safari ITP가 IndexedDB를 "침묵 차단"하는 메커니즘

IndexedDB는 이벤트 기반 API다.

```
indexedDB.open('mydb')
  → IDBOpenDBRequest 객체 반환
  → onsuccess 이벤트 발생 → DB 사용 가능
  → onerror 이벤트 발생 → 에러 처리
```

Safari ITP가 third-party iframe에서 이걸 차단할 때, **에러를 던지지 않는다.** 대신 `onsuccess`와 `onerror` 이벤트를 **아예 발화하지 않는다.** 이걸 Promise로 감싸면 `resolve()`도 `reject()`도 호출되지 않아 **영원히 pending 상태로 멈춘다.**

왜 에러를 안 던질까? Safari의 설계 원칙이다. 차단 사실을 알려주면 트래커가 우회를 시도할 수 있으므로, **"침묵하는 실패(silent failure)"**를 의도적으로 선택한 것이다.

##### 그런데 라이브러리도 같은 결론에 도달했다

흥미로운 점은, `flutter_secure_storage_web`의 **현재 최신 버전(v2.1.0)은 IndexedDB 대신 localStorage로 전환**했다는 것이다. 라이브러리 메인테이너도 같은 Safari iframe 문제를 인지하고 구조를 바꾼 것이다. localStorage는 동기 API라서 Safari ITP가 차단해도 hang이 아니라 빈 값을 반환하기 때문에, 적어도 앱이 멈추는 일은 없다.

우리는 v1.2.1(IndexedDB 기반)을 쓰고 있었기 때문에 hang이 발생했고, 라이브러리 업데이트 전에 인메모리 방식으로 직접 우회한 셈이다.

##### 플랫폼별로 FlutterSecureStorage가 쓰는 것

같은 `FlutterSecureStorage.write()`를 호출해도, 플랫폼에 따라 내부 동작이 완전히 다르다.

| 플랫폼         | 내부 구현                                          | 보안 수준                                 |
| -------------- | -------------------------------------------------- | ----------------------------------------- |
| **Android**    | Android Keystore(TEE) + EncryptedSharedPreferences | 하드웨어 레벨 — 키가 칩 안에서만 존재     |
| **iOS**        | Keychain + Secure Enclave                          | 하드웨어 레벨 — 물리적으로 분리된 보안 칩 |
| **Web (v1.x)** | WebCrypto(AES-GCM) + IndexedDB                     | 소프트웨어 레벨 — 키도 같은 저장소에 저장 |
| **Web (v2.x)** | WebCrypto(AES-GCM) + localStorage                  | 소프트웨어 레벨 — hang 문제는 해결        |

모바일에서는 OS가 제공하는 하드웨어 보안 모듈(Android TEE, iOS Secure Enclave)이 암호화 키를 보호한다. 키가 칩 밖으로 나오지 않기 때문에 루팅/탈옥해도 추출이 극히 어렵다.

반면 웹에는 이런 하드웨어 보안이 없다. `crypto.subtle`로 암호화해도 키 자체가 같은 브라우저 저장소에 저장되기 때문에, **금고 안에 금고 열쇠도 같이 넣어둔 셈**이다. `FlutterSecureStorage`라는 이름이 같아도, 웹에서의 "Secure"는 모바일과 근본적으로 다른 수준이라는 것을 이번에 배웠다.

#### 배운 점

- **점진적 마이그레이션에서 가장 위험한 지점은 "두 시스템 사이의 상태 동기화"다.** 각각은 잘 동작해도, 둘 사이의 통신이 빠지면 사용자는 깨진 경험을 하게 된다.
- **Safari의 ITP는 iframe 환경에서 스토리지 동작을 근본적으로 바꾼다.** `FlutterSecureStorage`처럼 내부 구현이 IndexedDB에 의존하는 라이브러리는 Safari iframe에서 예상과 다르게 동작할 수 있다. 라이브러리를 쓸 때 "웹에서 내부적으로 뭘 쓰는지"까지 확인하는 습관이 필요하다.
- **크롬에서만 QA하면 안 된다.** 이번 경험 이후 사파리 QA를 배포 체크리스트에 추가했다.
- **같은 API라도 플랫폼에 따라 보안 수준이 다르다.** 모바일의 "Secure"와 웹의 "Secure"는 하드웨어 보안 유무라는 근본적 차이가 있다. 라이브러리 이름만 보고 안심하면 안 된다.
